# Codex phase2 Gate Review

**Findings**

No high-severity issues found.

1. ISSUE, medium: `app/adapters/storage/firestore_backend.py:125-128`  
   `backfill_is_flagged()` is correct for small/offline migration, but it streams the whole collection and rewrites the whole document with `set({**data, ...})`. On a large/live collection, Firestore `stream()` has an RPC timeout caveat, and the full-document write can clobber concurrent changes. Safer shape: page by document ID and update/merge only `is_flagged`.

2. ISSUE, low: `firestore.indexes.json:11-38`, `firestore.indexes.json:55-70`  
   Index coverage is complete, but several composites are redundant. Firestore can support compound equality queries via automatic/index merging, and `get_items_for_run(digest_run_id + is_flagged)` has no `order_by`, so its composite is not required for correctness. This is storage/write-cost overhead, not a runtime bug.

3. ISSUE, low: `app/runner.py:58-64`  
   `_firestore_cache` is not locked. Concurrent first requests can both construct a client/backend before one wins the assignment. This is a benign duplicate-construction race, not a data leak.

Real snapshots expose `.id` and `.exists`; ordering is not guaranteed and duplicate refs may collapse, but this code returns a set, so that is fine. Fake parity at `tests/unit/fake_firestore.py:126-128` is adequate.

3. No Firestore pagination limit silently drops pages, but the single `stream()` approach has the documented stream timeout risk for large/slow scans.

4. 6. AGREE: Firestore caching is keyed by `("firestore", project)` at `app/runner.py:57-65`; SQLite remains per-call at `app/runner.py:53-56`.
