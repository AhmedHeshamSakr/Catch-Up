# Codex phase3 Gate Review

**Findings**

1. **HIGH ISSUE**: the Cloud Run product console cannot authenticate as documented. The backend now requires `API_KEY` before startup ([app/web_app.py:33](<app/web_app.py:33>)), and product routes like `/api/dashboard` require that key ([app/api/app.py:259](<app/api/app.py:259>)). But the static frontend only sends `X-API-Key` from build-time `NEXT_PUBLIC_API_KEY` ([frontend/lib/api.ts:49](<frontend/lib/api.ts:49>)), while the Dockerfile only sets `NEXT_PUBLIC_API_BASE` for the frontend build ([Dockerfile:26](<Dockerfile:26>)) and the README deploy command only sets runtime `API_KEY` ([README.md:279](<README.md:279>)). Result: `/api/health` works, but the deployed console will 401 on real data unless the image is built with `NEXT_PUBLIC_API_KEY`, which also means exposing the shared API key to browser JS.

2. **MEDIUM ISSUE**: `COPY frontend/ ./` after `npm ci` is too broad without a `.dockerignore` ([Dockerfile:25](<Dockerfile:25>)). This workspace has `frontend/node_modules`, `.next`, `out`, and `.env.local`; Docker does not use `.gitignore` by default. A local `docker build .` can copy host `node_modules` over the Linux builder’s freshly installed dependencies and can copy local env/build artifacts into the build stage.

3. **LOW ISSUE**: remaining doc drift outside the new Deployment section. README’s repo layout still says `Dockerfile` is the “container image for Cloud Run / Agent Engine” ([README.md:129](<README.md:129>)), but the Dockerfile is now explicitly the product image, not Agent Engine ([Dockerfile:15](<Dockerfile:15>)). `docs/ADK-GUIDE.md` also still says the deploy path is future ([docs/ADK-GUIDE.md:165](<docs/ADK-GUIDE.md:165>)).
