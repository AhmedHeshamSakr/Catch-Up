## 1. Verdict

The project is **not rotten — it is mis-mapped**. The code you actually wrote is, in most subsystems, genuinely good-to-excellent (the pipeline, LLM layer, storage port, collectors, infra services, API internals, and the 497-test suite are all real, disciplined, and well-engineered). The "scattered" feeling is correct, but it lives almost entirely at the **seams**: the spec/ARCHITECTURE docs promise a hexagonal tree, 8 screens, and a "swap-by-config" GCP prod path that the code only half-delivers, while the actual deploy artifact (`fast_api_app.py` + Dockerfile) is a third, disconnected story that ships no console and ephemeral storage. The single biggest problem is **the deployment + product-surface incoherence**: three competing narratives (docs say full GCP prod; manifest says `deployment_target: none`; Docker ships API-only with no UI) plus two ASGI entrypoints, which is exactly the "agents vs gcp webapp vs sqlite are all scattered" sensation. Production-readiness grade: **C** — the local desktop app genuinely works end-to-end and is well-tested, but no CI, an open-by-default API that ships into the container, an unproven Firestore path, and a deploy artifact that contradicts its own docs keep it well short of production-grade. As a *local v1*, it is a solid **B**; as the *cloud product the docs advertise*, it is a **D**.

## 2. Is it scattered? (the honest answer)

**Your instinct is right, but you're blaming the wrong layer.** The individual files are not scattered — they're some of the cleanest code in any hobby repo I've reviewed. What's scattered is the **map-vs-territory gap** and three specific seams.

**The agents — feels scattered, is actually coherent.** The ADK tree (`app/pipeline/agents.py`) exactly matches the documented 7-stage diagram, propagates state correctly via `EventActions.state_delta` so it survives a persistent session service, has a single source of truth for source types (`COLLECTED_SOURCE_TYPES`), and a correct fail-closed critic/reflection loop. 87 pipeline tests pass. **The agents are the strongest part of the system.** Why they *feel* disconnected: there is **no Pipeline screen** (spec screen 6), so the guardrail's audit output (`run.flagged`, `run.critic_verdicts`) is computed, persisted, even serialized by the API — then silently dropped by the frontend Zod schema. You built smart agents and gave the user no window into them. That's a missing surface, not broken plumbing.

**The GCP/web deployment story — genuinely fragmented.** This is the real scatter. There are **two ASGI entrypoints**: `app.api.app:create_app` (desktop — console + API + scheduler + durable sessions, port 8000) and `app.fast_api_app:app` (Docker — ADK web UI + bare `/api/*`, in-memory ADK sessions, **no console**, port 8080). The Dockerfile copies only `app/` and `config/`, never builds or ships `frontend/out`, so the "GCP webapp" is a backend with no UI. The `agents-cli-manifest.yaml` still carries stock scaffold defaults (`deployment_target: none`, `session_type: in_memory`, `datastore: none`, `region: us-east1`) that contradict both the README's GCP prose and the project's own `global` location guidance. **Three documents describe three different products.** This is the concrete, evidence-based core of your complaint.

**The SQLite/Firestore seam — half-real.** This is the *honorable* part of the scatter. Storage is the **one place** that fully honors the intended hexagon: a clean `StorageBackend` ABC, two adapters, a config-driven `build_storage()` factory, and a single shared contract test subclassed by both backends. SQLite is genuinely production-grade for v1 (WAL, busy_timeout, online migrations, indexes). **But Firestore is vestigial**: tested only against an in-memory fake, the real emulator gate is permanently skipped and raises `AssertionError`, it uses deprecated positional `where()`, has no composite indexes, does O(N) document gets for dedup, and has a real `is_flagged`-missing-field data-loss divergence from SQLite's NULL handling. Setting `STORAGE_BACKEND=firestore` in prod would pass every offline test, then throw `FAILED_PRECONDITION` on the first filtered news query. Multi-tenant `org_id`/`user_id` is decorative — stored but never used in any WHERE clause; `user_id` isn't even a SQLite column.

**Net:** the agents and storage *files* are coherent. The deployment surface is genuinely fragmented. The "scattered" feeling is real and justified — it's just structural (missing screens, missing ports, contradictory deploy descriptors), not rot.

## 4. Critical & High Issues (must-fix)

### Security (fix first)
- **API is open-by-default and the default ships into the deployed container.** `config.py:138` defaults `api_key=None`; `app.py:104-105` makes `_require_api_key` a no-op when unset; `fast_api_app.py:77` mounts all `/api/*` onto the Cloud Run image. Deployed without `API_KEY` (nothing sets it), the data routes are world read+write: overwrite sources/watchlist, trigger runs, read all news. **Fix:** fail closed — refuse to start when bind is non-loopback and `api_key` is unset; require `API_KEY` at import in `fast_api_app.py`. *(Settings routes are separately loopback-guarded, so that subset is safe.)*
- **SSRF guard has no response-size cap.** `app/services/net.py:74-132` enforces scheme/private-IP/IP-pin/timeout but `client.send()` is non-streamed; every caller reads the full body into memory. A user-supplied URL (`POST /sources/resolve`) returning a multi-GB body from a legitimate public host OOMs the process. **Fix:** stream and abort past a 5-10 MB cap; reject on oversized `Content-Length`.
- **Live Google AI Studio + GNews keys in plaintext in `app/.env`.** Gitignored and never committed, but live and billable. **Fix:** rotate both keys; add a gitleaks pre-commit/CI scan.

### Deployment / process
- **No CI anywhere.** No `.github`/`.gitlab`/`.circleci`, no pre-commit; manifest `cicd_runner: skip`. 497 real tests + the eval regression gate run only when a human types the command. **Fix (highest-leverage):** one `.github/workflows/ci.yml` running `uv run pytest tests/unit tests/integration`, `cd frontend && npm ci && npm run test && npm run build`, and optionally the offline eval gate. Do this *before* touching anything else — it protects every subsequent change.
- **Deploy descriptors contradict the docs / no UI in the container.** Dockerfile never builds `frontend/out`; `fast_api_app.py` never calls `mount_console`; manifest says `deployment_target: none`. The "GCP webapp" is API-only. **Fix:** pick ONE — either add a Node build stage + `mount_console` so Cloud Run is single-port like desktop, OR document the container as API-only and stop the prose/comment from promising a deployed console.

### Architecture
- **Layering inversion / import cycle.** `app/pipeline/agents.py:47-54` imports `_collect`, `_default_*`, `select_rendered` from `app/runner.py` at top level; `runner.py` imports pipeline back (deferred to dodge the cycle). **Fix:** move `_collect` and the default factories down into `app/pipeline/` so the pipeline imports downward only.
- **Import-time side effects.** Importing *any* `app.*` submodule runs `app/__init__.py:15 → app/agent.py:24-26`, which builds a pipeline and **creates/migrates a SQLite file on disk** (verified: `import app.core.domain` materializes `data/catchup.db`). Crashes on read-only FS; pollutes test collection; the import-time `root_agent`/`run_id` are thrown away by the real run path. **Fix:** make `root_agent` a lazy factory; drop the agent import from `app/__init__.py`; no `init_schema()` at import.

### Incompleteness (high-impact gaps)
- **Pipeline screen (UI 6) entirely missing.** No agent tree, no per-agent config, no traces, and the critic/guardrail audit trail has nowhere to land. The single most spec-divergent surface, and the direct cause of "agents feel disconnected." **Fix:** build a minimal read-only Pipeline screen (agent tree + per-run `flagged`/verdicts + global model/temp editor) OR formally drop screen 6 from the spec. Don't leave it as a silent gap.
- **Firestore would fail on first real query.** (See §2.) **Fix:** either fund it (un-skip emulator test, `FieldFilter`, composite indexes, batched `existing_ids`, `is_flagged` backfill, client caching) or mark it experimental in README and drop it from the "config-only swap" marketing.

## 9. Codex Independent Review — Corrections & Additions

An independent `codex-cli` review (read-only) fact-checked this report against the live code. Full review: `.claude/codex-reviews/20260621T170255Z-deep-audit-review.md`. Verdict: *"mostly trustworthy on direction and prioritization, but not always calibrated."* It confirmed every load-bearing claim (open-by-default API, two ASGI entrypoints, import-time SQLite side effects, layering inversion, render output-key mismatch, no CI, SSRF size-cap gap) via file:line. Adopt these calibrations:

**Overstatements to narrow (this report was too strong):**
- **Firestore "would fail on first real query" → too broad.** Doc reads/writes and `list_runs()` likely work; it's the *filtered+ordered* `list_news()` that breaks without composite indexes. Also README + `docs/ADK-GUIDE.md:158` already caveat Firestore as not live-validated, so it isn't silently oversold. Restate as: *"filtered Firestore news queries fail without composite indexes; the backend is not live-validated."*
- **"Docker ships no UI" → say "no Next.js product console."** The ADK web UI *is* enabled (`fast_api_app.py:62-68 web=True`); what's missing is the product console (`frontend/out` is never built/copied).
- **"Live/billable keys" → not provable statically.** Restate as "key-shaped plaintext secrets present on disk." Rotation still warranted.
- **Per-agent config "entirely missing" → too broad.** A `judge_model` override exists for offline eval (`config.py:88-92`, `judge.py:60-68`); it's the *runtime pipeline stages* that share one global model/temp.
- **Render output-key mismatch impact → display badges, not downloads.** Files aren't served either way (`output-links.tsx:43-45` says so explicitly); fixing the key set restores the format badges, not download capability.
- **Guardrail data "has nowhere to land" → API already serializes it.** `RunDetail`/`DigestRun` (`schemas.py:17-19`) includes `flagged`/`critic_verdicts`; the loss is purely at the frontend Zod schema (`schemas.ts:59-72`). (Doesn't change the fix.)

**Net-new issues Codex found that this audit MISSED (add to the backlog):**
- **[high] Cloud/Docker Firestore can't even import as shipped.** `google-cloud-firestore` is an *optional* extra (`pyproject.toml:42-43`) but the Dockerfile runs plain `uv sync --frozen` (`Dockerfile:29`) — so `STORAGE_BACKEND=firestore` fails at import, *before* the index problem. The Firestore swap is more broken than this report said.
- **[medium] Local CSRF / run-trigger gap.** With auth open by default, `POST /api/runs` (`app.py:273-281`) needs no body/header, so a malicious web page can trigger a local digest run (CORS blocks reading the response, not the side effect). Product routes lack the Origin/Host loopback checks the settings surface has.
- **[medium] Unauthenticated `/feedback` on the deployed entrypoint.** `fast_api_app.py:80-91` writes arbitrary `text` (`app_utils/typing.py:26-34`) to Cloud Logging with no API key and no rate limit — log-injection / cost vector.
- **[note] `NEXT_PUBLIC_API_KEY` is not public-user auth.** `api.ts:49-57` ships the key to the browser; fine for trusted single-user/internal use, but the API-key control is a *stopgap*, not a public web-auth model. Frame it that way in any "secure the API" work.
